# AI 智能匹配与能力图谱系统生产化改造软件设计书

## 1. 文档目的

本文档用于指导当前系统从“可演示版本”升级为“可持续迭代、可上线交付版本”，覆盖以下三项改造目标：

1. 将前端替换为 `index/` 目录中的新设计方案。
2. 优化后端与数据库结构，降低 Mock / Legacy 逻辑对真实业务的干扰。
3. 清理冗余文件与无用代码，形成更清晰的工程结构和更稳定的交付基线。

## 2. 当前问题概述

### 2.1 前端问题

- 当前正式前端位于 `frontend/`，新设计稿位于 `index/`，形成双前端并存状态。
- 两套前端目录结构高度相似，但依赖、图谱实现、静态资源、动画能力不同，导致维护成本高。
- 前端默认仍保留 Mock 能力，存在误用假数据的风险。
- 仓库中残留测试文件、重复资源、旧实现目录，容易误导后续开发。

### 2.2 后端问题

- 当前后端业务主要集中在 `InMemoryDataService`，服务职责过大，边界不清晰。
- 登录 Token 仅为可推导字符串，未签名、无时效、无防篡改能力。
- 管理端接口缺少真正的管理员鉴权。
- 控制器中夹杂 Prompt、Mock 返回、Legacy 兼容逻辑，降低可维护性。
- 部分接口返回 demo 数据，不利于生产场景的真实可解释性。

### 2.3 数据库问题

- 当前 KingbaseES 已可真实连接，但数据库约束仍偏弱。
- 用户手机号/邮箱缺少库级唯一约束。
- 人才投递关系缺少唯一约束，存在重复投递记录风险。
- 种子数据仍使用明文密码，不符合生产要求。

## 3. 改造目标

### 3.1 业务目标

- 前端只保留一套正式工程入口。
- 所有主要页面默认对接真实后端接口。
- 登录、管理、文档、匹配、消息功能基于真实数据库运行。
- 形成可以继续扩展的后端边界与数据库约束体系。

### 3.2 技术目标

- 前端：统一到 `frontend/`，吸收 `index/` 的新设计与资源。
- 后端：保留 Spring Boot 单体结构，但收紧认证、配置与控制器职责。
- 数据库：保留 Flyway + KingbaseES（PostgreSQL 兼容协议）路线，补齐关键约束和迁移。
- 工程：删除重复实现、无用测试样本、旧前端目录与显式 Mock 依赖。

## 4. 目标架构

### 4.1 前端架构

- 正式前端目录：`frontend/`
- 设计来源目录：`index/`
- 替换策略：
  - 以 `index/` 为视觉与交互基线整体替换 `frontend/`
  - 保留原有 API 契约与路由结构，确保无需大面积改动后端接口
  - 图谱组件统一到新前端当前采用的实现方案

### 4.2 后端架构

- 保持单体 Spring Boot 架构，按职责收口为以下模块：
  - `auth`: 登录、Token 签发、请求用户解析
  - `document`: 文档上传、解析、原文权限校验
  - `match`: 岗位推荐、候选人推荐、匹配详情
  - `chat`: 会话、消息、已读状态
  - `admin`: 管理员用户管理
- 本轮不做大规模包重构，但先完成以下“上线化收口”：
  - Token 改为签名 Token
  - 新增统一鉴权辅助
  - 管理端接口必须管理员登录
  - 删除明显 Mock / Legacy 分支

### 4.3 数据库架构

- 数据库产品：KingbaseES
- 连接方式：PostgreSQL 兼容协议
- 迁移工具：Flyway
- 本轮重点增强：
  - 用户手机号唯一索引
  - 用户邮箱唯一索引
  - 人才投递唯一约束
  - 明文密码逐步淘汰

## 5. 实施方案

### 5.1 前端替换方案

1. 用 `index/` 内容覆盖 `frontend/` 的页面、布局、静态资源与依赖声明。
2. 保留真实接口基址配置 `VITE_API_BASE`。
3. 去除默认 Mock 依赖，改为真实接口优先。
4. 替换完成后删除以下冗余：
   - `index/` 目录
   - 旧测试样本文件
   - 不再引用的前端资源与组件

### 5.2 后端优化方案

1. 登录改为签名 Token：
   - Token 中包含 `userType`、`account`、`issuedAt`、`expireAt`
   - 使用服务端密钥做 HMAC 签名
   - Bearer 解析必须校验签名与有效期
2. 增加统一鉴权辅助：
   - `optionalUser`
   - `requireUser`
   - `requireAdmin`
3. 管理端接口改为强制管理员访问。
4. 去除明显 Legacy / Mock 逻辑：
   - 前端 API Mock 默认关闭并逐步移除
   - 匹配详情 Legacy ID 分支下线
   - 文档结果中的 demo 返回清理
5. 密码策略优化：
   - 继续使用 BCrypt
   - 对历史明文密码提供透明升级或迁移策略
   - 管理员重置密码接口不再回传默认密码明文

### 5.3 数据库优化方案

新增 Flyway 迁移脚本，完成：

- `aimap_user(user_type, phone)` 唯一索引
- `aimap_user(user_type, email)` 唯一索引
- `aimap_talent_pool(document_id, target_company_user_type, target_company_account)` 唯一约束
- 按查询场景补充索引

## 6. 冗余清理范围

本轮计划删除或清理：

- 双前端并存产生的 `index/` 目录（在替换完成后）
- `frontend/tmp/test.pdf`
- 不再引用的 Mock 代码
- 不再引用的旧图谱/旧页面资源
- 后端中已无业务价值的 Legacy 分支

## 7. 风险与回滚

### 7.1 风险

- 新前端整体替换可能带来依赖差异和样式回归。
- Token 机制升级后，旧登录状态会失效，需重新登录。
- 唯一索引迁移前若存在脏数据，会导致迁移失败。

### 7.2 回滚策略

- 前端替换前保留 Git 工作区记录，可回退到替换前状态。
- Token 升级若发现兼容问题，可短时保留旧解析兼容层。
- 数据库迁移执行前先检查重复数据，必要时先清洗再执行。

## 8. 验收标准

### 8.1 前端验收

- `frontend` 能正常安装依赖并构建成功。
- 登录页、个人端、企业端、管理端页面样式切换为新设计。
- 文档、匹配、图谱、消息、管理页可正常访问。

### 8.2 后端验收

- 登录返回签名 Token。
- 非管理员无法访问管理员接口。
- 文档、匹配、消息接口可在真实 KingbaseES 上正常工作。
- 不再返回明显 demo / mock 数据。

### 8.3 数据库验收

- Flyway 迁移成功。
- 关键唯一约束生效。
- 后端可在真实 KingbaseES 上稳定启动并完成接口验证。

## 9. 本轮实施顺序

1. 编写设计书
2. 替换 `frontend` 为 `index` 新前端
3. 去除前端默认 Mock 依赖
4. 升级后端 Token 与管理员鉴权
5. 补充数据库迁移与密码策略优化
6. 删除冗余文件与旧目录
7. 执行前后端构建、数据库联调与接口验证
