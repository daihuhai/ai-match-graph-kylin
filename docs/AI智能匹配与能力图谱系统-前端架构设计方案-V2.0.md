## 1. 前端目标与边界

### 1.1 目标
在“LoongArch + 银河麒麟 V11”的信创部署环境约束下，前端以 **静态资源部署** 为主，通过 **Vue 3 + Element Plus + AntV G6** 实现：
- 双角色（个人 / 企业）端到端闭环：注册登录 → 文档上传 → 解析进度 → 图谱查看 → 匹配推荐 → 反馈；
- 管理端：用户 / 文档 / 技能库 / 匹配记录 / 监控面板；
- 图谱可视化与交互：力导向 / 层次 / 辐射等布局，支持拖拽、缩放、搜索、节点展开收起；
- 匹配结果解释：匹配度评分、技能差距与建议可视化（雷达图、条形图等）。

### 1.2 前端边界（与后端协作约定）
前端不直接执行 OCR/LLM/图谱推理/向量检索，仅负责：
- 文件上传与状态轮询/推送；
- 结果展示、交互与“可解释性”UI（证据来源、差距项、学习路径展示）；
- 权限控制与审计 UI（展示日志/监控，不直接采集系统指标）。

## 2. 总体前端架构与技术选型

### 2.1 架构分层（前端视角）
前端采用“**UI 表现层** + **业务模块层** + **基础设施层**”的轻量化架构：
- **UI 表现层**：页面（Views）+ 通用组件（Components）；
- **业务模块层**：按领域拆分（auth、document、graph、match、admin），沉淀业务组件与 hooks；
- **基础设施层**：路由、状态管理、API 客户端、权限守卫、日志与埋点（可选）、工具库。

### 2.2 技术栈（建议落地版本）
| 类别 | 组件 | 推荐版本 | 用途 |
|---|---|---:|---|
| 框架 | Vue | 3.4.x | 组合式 API、性能与生态 |
| 构建 | Vite | 5.x | 构建快、配置轻量 |
| 语言 | TypeScript | 5.x | 提升可维护性与契约性 |
| UI | Element Plus | 2.7.x | 组件覆盖全面、适配企业后台 |
| 图谱 | AntV G6 | 5.0.x | 图布局与交互能力强 |
| 图表 | ECharts | 5.5.x | 雷达/柱状/饼图等 |
| 状态管理 | Pinia | 2.1.x | 轻量、与 Vue3 匹配 |
| 路由 | Vue Router | 4.3.x | 页面路由与守卫 |
| HTTP | Axios | 1.7.x | 拦截器、错误统一处理 |
| CSS | UnoCSS（可选） | 0.58.x | 原子化按需生成，控制体积 |
| 代码规范 | ESLint + Prettier | latest | 统一风格 |

### 2.3 信创环境适配原则（前端）
前端以浏览器为运行时，主要风险在于：
- 信创浏览器对现代特性的支持差异（ES2020+、WebGL、字体渲染等）；
- G6 依赖 WebGL/Canvas，需保证显卡驱动与浏览器 WebGL 可用。

因此前端策略：
1. 构建时降级输出：控制目标语法（`build.target`）、必要 polyfill；
2. 运行时能力探测：WebGL 检测，不满足则降级为 Canvas 或简化渲染；
3. 资源体积控制：避免过多依赖引入，启用按需加载与分包。

## 3. 前端工程结构与规范

### 3.1 目录结构（建议）
```txt
src/
  assets/                 # 图片、字体、全局样式
  components/             # 通用组件（表格、弹窗、上传组件封装等）
  layouts/                # 布局（双角色主布局、管理端布局）
  views/                  # 页面（按路由）
    auth/
    person/
    company/
    admin/
  modules/                # 业务模块（领域逻辑聚合）
    auth/
    document/
    graph/
    match/
    admin/
  router/                 # 路由与权限守卫
  stores/                 # Pinia store
  api/                    # API 客户端封装（axios 实例、接口）
  types/                  # TS 类型（DTO/VO、Graph 数据结构等）
  utils/                  # 工具（校验、格式化、缓存等）
  hooks/                  # 可复用 hooks（轮询、分页、权限等）
  main.ts
  App.vue
```

### 3.2 约定与规范
- **命名**：组件 PascalCase；文件夹 kebab-case；store `useXxxStore`；
- **业务模块拆分优先**：避免所有逻辑堆在 views；
- **类型先行**：与后端 DTO/VO 形成对应关系，关键接口类型必须落到 `src/types`；
- **错误统一处理**：Axios 响应拦截器把错误规范化，页面只处理“业务态”。

## 5. 路由、布局与权限体系（RBAC）

### 5.1 路由分组
建议按角色与端拆分路由：
- `/auth/*`：登录、注册；
- `/person/*`：个人端；
- `/company/*`：企业端；
- `/admin/*`：管理端（仅管理员）。

### 5.2 布局策略
- `AuthLayout`：极简布局；
- `MainLayout`：个人/企业端通用（顶部用户信息 + 侧边菜单 + 内容区）；
- `AdminLayout`：管理端（多级菜单、面包屑、标签页可选）。

### 5.3 权限模型（前端）
后端使用 Sa-Token RBAC，前端配合：
1. 登录后获取 `token` + `userType`（person/company/admin）+ `permissionCodes`（可选）；
2. 路由守卫进行：
   - 未登录 → 跳转登录；
   - 已登录但角色不匹配 → 403 页面；
   - 细粒度权限（按钮级）→ `v-permission` 指令或组合式函数控制显示/禁用。

#### 5.3.1 路由守卫伪代码
```ts
router.beforeEach(async (to) => {
  const auth = useAuthStore()
  if (to.meta.public) return true
  if (!auth.token) return { name: 'Login', query: { redirect: to.fullPath } }
  if (to.meta.userTypes && !to.meta.userTypes.includes(auth.userType)) return { name: 'Forbidden' }
  return true
})
```

### 5.4 Token 存储建议
在信创内网环境下仍需防范 XSS：
- 优先：token 存内存 + 刷新机制（需要后端支持 refresh token）；  
- 可接受：`sessionStorage`（关闭浏览器即失效）；  
- 不建议：长期 `localStorage` 明文保存。

### 6.1 API 规范对齐（来自总体方案）
- 基础路径：`/api/v1/`

### 6.2 Axios 实例与拦截器（建议）
```ts
// src/api/http.ts
import axios from 'axios'
import { useAuthStore } from '@/stores/auth'

export const http = axios.create({
  baseURL: import.meta.env.VITE_API_BASE,
  timeout: 15000,
})

http.interceptors.request.use((config) => {
  const auth = useAuthStore()
  if (auth.token) config.headers.Authorization = `Bearer ${auth.token}`
  return config
})

http.interceptors.response.use(
  (resp) => {
    const body = resp.data
    if (body?.code !== 200) return Promise.reject(body)
    return body.data
  },
  (err) => Promise.reject(err),
)
```

## 7. 文档上传与解析进度前端方案

### 7.1 上传策略
支持 DOC/PDF，限制 50MB：
- 前端校验：文件后缀 + MIME（仅作提示，最终以后端校验为准）；
- 采用分片上传：**可选项**（后端支持则开启；若不支持则使用普通上传）；
- 上传完成后返回 `docId`，进入解析任务跟踪。

### 7.2 解析任务跟踪（轮询方案）
在资源受限单体后端中，优先使用轮询（而非 WebSocket）：
- 轮询接口：`GET /api/v1/document/{id}/status`
- 轮询间隔：1s（处理中）→ 3s（排队中）→ 10s（长时间未完成）自适应退避；
- 终止条件：DONE/FAILED。

建议封装 `usePolling`：
```ts
// src/hooks/usePolling.ts
export function usePolling<T>(fn: () => Promise<T>, intervalMs = 2000) {
  let timer: number | null = null
  const start = () => { stop(); timer = window.setInterval(fn, intervalMs) }
  const stop = () => { if (timer) window.clearInterval(timer); timer = null }
  return { start, stop }
}
```

### 7.3 解析结果展示
接口：`GET /api/v1/document/{id}/result`  
前端展示策略：
- 原始 JSON：提供“开发模式”折叠面板（便于联调）；
- 结构化表单：按“技能/经历/教育/项目”等 Tab 分区；
- 证据链：展示“证据来源”（文档页码/段落，若后端提供）。

## 8. 能力图谱可视化方案（AntV G6 5.x）

### 8.1 图谱数据接口与数据格式
建议后端输出 G6 友好的结构（nodes/edges），减少前端转换成本：
```ts
// src/types/graph.ts
export interface GraphNode {
  id: string
  label: string
  type: 'Person' | 'Job' | 'Skill' | 'Project' | 'Company' | 'Category'
  props?: Record<string, any>
}

export interface GraphEdge {
  id: string
  source: string
  target: string
  type: 'HAS_SKILL' | 'REQUIRES_SKILL' | 'DEPENDS_ON' | 'BELONGS_TO' | 'APPLIED_IN' | 'PUBLISHES'
  props?: Record<string, any>
}

export interface GraphData {
  nodes: GraphNode[]
  edges: GraphEdge[]
}
```

推荐接口：
- `GET /api/v1/graph/person/{id}` → 个人能力子图；
- `GET /api/v1/graph/job/{id}` → 职位能力子图；
- `POST /api/v1/graph/search` → 节点搜索与定位（返回 nodeId 或子图）。

### 8.2 视觉编码（样式规范）
| 节点类型 | 形状 | 主色 | 说明 |
|---|---|---|---|
| Person | 圆形/头像节点 | #1677ff | 个人中心 |
| Job | 圆角矩形 | #fa8c16 | 职位中心 |
| Skill | 圆形 | #52c41a | 技能点 |
| Project | 菱形 | #722ed1 | 项目经验 |
| Company | 矩形 | #13c2c2 | 企业实体 |
| Category | 标签/胶囊 | #8c8c8c | 分类体系 |

边样式建议：
- `REQUIRES_SKILL`：实线；必需加粗/红边；加分项虚线；
- `DEPENDS_ON`：箭头方向明确，颜色更浅；
- hover 高亮：聚焦 1-hop 邻居，其它降低透明度。

### 8.3 布局与交互
提供布局切换：
- Force（全局浏览/聚类发现）
- Dagre/Hierarchy（分类体系/依赖链）
- Radial（以人/职位为中心）

核心交互：
- 拖拽画布、节点拖拽；
- 滚轮缩放；
- 框选/点选；
- 节点详情抽屉（props 展示 + 关联列表）；
- 搜索定位：输入技能/节点名 → 高亮并自动居中；
- 节点展开/收起：对超千节点建议“按层加载”。

### 8.4 大图性能策略（千级节点）
1. **服务端裁剪**：默认只返回 1~2 跳子图；  
2. **增量加载**：点击节点再加载邻居（lazy expand）；  
3. **渲染降级**：超过阈值时：
   - 关闭阴影/复杂渐变；
   - 文本仅在一定缩放级别显示；
   - 使用聚合（cluster）/折叠（collapse）节点；  
4. **离屏与节流**：resize、hover、search 高亮均需节流。

### 8.5 WebGL 可用性检测与降级
若目标浏览器 WebGL 不可用：
- 降级为 Canvas 渲染；
- 限制布局为层次/辐射，减少 force 迭代；
- 降低节点数量（强制更小子图）。

## 10. 性能、兼容性与安全加固（信创浏览器）

### 10.1 性能目标（前端侧）
- 首屏可交互：≤ 2s（内网）；
- 图谱渲染：千级节点可用（交互不卡死）；
- 匹配页面：分页查询/筛选响应流畅。

### 10.2 构建优化
- 路由级懒加载：`import()`；
- 依赖分包：将 `g6`、`echarts` 独立 chunk；
- 图片/字体压缩与缓存策略；
- 关闭生产 sourcemap（或仅内网保留）；
- 组件按需：Element Plus 按需引入。

### 10.3 兼容性策略
- Vite 构建 target 降级（示例：`es2018` 或更低，视信创浏览器内核而定）；
- 必要 polyfill（如 Promise、URL、AbortController 视情况引入）；
- CSS 兼容：避免过新的选择器/特性，必要时加 postcss 处理；
- WebGL 检测与降级（见 8.5）。

### 10.4 安全建议
- 前端输入校验（但不替代后端校验）；
- 防 XSS：不使用 `v-html` 渲染不可信内容；如需渲染富文本，必须做 sanitize；
- 防 CSRF：如采用 token header 方式，默认风险较低；仍建议后端校验 Origin/Referer；
- 文件上传：仅显示允许类型，前端不解析文件内容（避免攻击面扩大）。

## 11. 构建、离线依赖与部署（Nginx）

### 11.1 环境变量
建议通过 `.env.*` 管理：
```bash

VITE_APP_TITLE=AI智能匹配与能力图谱系统
```

### 11.2 Nginx 部署（静态资源 + 反向代理）
前端最终交付为 `dist/`，部署到银河麒麟服务器 Nginx：
```nginx
server {
  listen 80;
  server_name _;

  root /opt/app/front/dist;
  index index.html;

  location / {
    try_files $uri $uri/ /index.html;
  }

  # 反向代理后端（单体 Spring Boot）
  location /api/ {
    proxy_pass http://127.0.0.1:8080;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
  }
}
```

### 11.3 离线部署注意事项（前端）
由于离线环境可能缺少 Node 生态依赖下载能力：
- 推荐方式 A：在有网环境完成 `pnpm install && pnpm build`，将 `dist/` 拷贝到离线服务器；
- 推荐方式 B：准备离线 npm 仓库/制品库（如 Verdaccio）+ 离线依赖包；
- Node 在 LoongArch 可用性需提前验证（若不可用，优先采用方式 A）。

| parse_fail_top_reason | 失败原因 Top | 条形图/列表 | 60s |
| match_avg_ms | 匹配平均响应 | 数值 + 趋势 | 30s~60s |
| model_mode | 本地/云端 | 状态 Tag | 10s~30s |
| service_health | 服务健康状态 | 红黄绿 | 10s~30s |

> 若后端暂未提供“监控聚合接口”，前端可先按 mock 结构预留 UI，占位不影响评审（评审更看重“有监控能力”）。

### 15.3 日志审计与导出（对标“基本运营”）
管理端建议提供：

## 18. 后端总体架构增补方案（在前端不变前提下）

本章节在不改动既有前端页面、路由、交互与接口消费习惯的前提下，补齐后端能力，使系统可实现“可运行、可演示、可验收”的完整闭环。

### 18.1 设计原则
- 前端接口契约优先复用：沿用现有 `/api/v1/*` 语义，避免前端改造成本。  
- 服务分层清晰：Controller（接口层）→ Service（业务层）→ Repository（数据层）→ Infra（对象存储/模型网关/缓存）。  
- 先可用、再增强：先满足赛题基本功能，再扩展优化（异步任务、缓存、重排模型、监控告警）。  
- 与银河麒麟环境一致：后端运行于 LoongArch + 银河麒麟 V11/V10，优先选用稳定中间件组合。  

### 18.2 推荐后端技术栈（Spring Boot 主线）
| 类别 | 推荐 | 说明 |
|---|---|---|
| Web 框架 | Spring Boot 3.x | 与当前方案一致，便于团队协同 |
| 鉴权 | Spring Security + JWT | 适配双角色与管理端 RBAC |
| ORM | MyBatis-Plus / JPA（二选一） | 快速实现业务 CRUD 与分页 |
| 数据库 | 国产关系型数据库（openGauss / 达梦 / TiDB 国产发行版） | 满足赛题“国产数据库”约束 |
| 缓存 | Redis | 会话、热点匹配结果、限流计数 |
| 对象存储 | MinIO（内网） | 文档原件、解析中间产物 |
| 消息队列（可选） | RabbitMQ / RocketMQ | 文档解析异步化与重试 |
| 搜索（可选） | Elasticsearch / OpenSearch | 职位和技能检索增强 |
| 观测 | Micrometer + Prometheus + Grafana | 性能与健康监控 |

### 18.3 逻辑架构（后端）
1. `auth-service`：注册登录、JWT 签发、RBAC 校验。  
2. `document-service`：上传、存储、解析任务编排、结构化结果落库。  
3. `graph-service`：能力图谱构建、查询、子图检索、图数据聚合。  
4. `match-service`：双向匹配计算、评分解释、推荐排序、反馈学习。  
5. `admin-service`：用户/文档/技能词典/审计日志/监控聚合接口。  
6. `llm-gateway`：统一接入外部大模型 API（含提示词模板、重试、审计、降级）。

## 19. 后端功能需求实现清单（覆盖基本功能）

### 19.1 用户与权限（个人/企业/管理员）
- 支持个人、企业注册与登录，密码加盐哈希存储。  
- 登录成功返回 `accessToken + refreshToken + userType + permissions`。  
- 个人仅可管理本人资料与文档；企业仅可管理本企业职位与候选人筛选；管理员拥有平台管理权限。  
- 提供权限审计字段：`operatorId`、`resourceType`、`resourceId`、`action`、`timestamp`。  

### 19.2 文档上传与解析
- 支持 `pdf/doc/docx` 上传；后端二次校验 MIME 与大小（建议 50MB）。  
- 上传即创建解析任务：`PENDING -> PARSING -> SUCCESS/FAILED`。  
- 解析产出结构化字段：教育、经历、技能、证书、项目、岗位要求等。  
- 返回证据片段（原文位置/句段）用于前端“可解释展示”。  
- 支持失败重试与错误码回传（如 `PARSE_TIMEOUT`、`FORMAT_UNSUPPORTED`）。  

### 19.3 能力图谱服务
- 图谱实体：`Person`、`Company`、`Job`、`Skill`、`Experience`、`Certificate`。  
- 图谱关系：`HAS_SKILL`、`REQUIRES_SKILL`、`SIMILAR_SKILL`、`WORKED_AT`。  
- 提供个人图谱、职位图谱、关键词搜索定位、邻域扩展接口。  
- 保证返回结构与前端 G6 数据模型兼容（`nodes/edges/meta`）。  

### 19.4 双向匹配与推荐
- 个人端：根据候选人画像推荐岗位列表。  
- 企业端：根据岗位需求推荐候选人列表。  
- 匹配评分至少包含：技能匹配、经验匹配、教育匹配、地域匹配（可选）等分项。  
- 返回解释字段：缺失技能清单、等级差距、建议学习路径。  
- 支持反馈闭环：收藏/不感兴趣/人工纠正，作为排序调整信号。  

### 19.5 管理后台能力
- 用户管理：查询、禁用/启用、角色变更。  
- 文档管理：任务状态检索、失败任务重跑、异常统计。  
- 技能词典管理：技能标准词、同义词、停用词维护。  
- 监控能力：接口耗时、错误率、任务积压、模型调用成功率。  
- 审计日志：关键操作可追溯，支持按时间、用户、模块过滤导出。

## 20. 后端非功能需求实现清单（稳定性/安全/性能）

### 20.1 性能与容量目标（基础版）
- 普通查询接口 P95 响应时间 <= 500ms。  
- 匹配详情/图谱接口 P95 响应时间 <= 1200ms。  
- 文档解析任务平均完成时长 <= 60s（与文档大小相关，需在测试报告注明样本）。  
- 并发目标（基础演示版）：50~100 并发用户下核心流程可用。  

### 20.2 稳定性目标
- 服务可用性目标 >= 99.0%（比赛演示阶段）。  
- 关键任务具备重试机制（指数退避 + 最大重试次数）。  
- 关键依赖超时配置明确（数据库、对象存储、LLM API）。  
- 统一错误响应格式，避免前端出现不可识别异常。  

### 20.3 安全与合规
- JWT + RBAC + 接口鉴权白名单机制。  
- 输入校验与输出脱敏（手机号、身份证、邮箱等敏感信息）。  
- 上传文件进行扩展名、MIME、恶意内容检测（至少做基础校验）。  
- 全链路 HTTPS（内网可通过网关 TLS 终止）。  
- 审计日志不可随业务删除，满足追溯要求。  

### 20.4 可运维性
- 提供 `health/readiness/liveness` 健康检查端点。  
- 指标上报：QPS、错误率、P95、队列长度、任务成功率。  
- 关键链路日志具备 `traceId`，便于定位跨服务问题。

## 22. 大模型 API 接入方案（云端调用）

按你的要求采用“调用大模型 API”模式，不在浏览器端直接调用，由后端统一代理。

### 22.1 接入原则
- 前端不保存任何模型密钥；仅调用后端业务接口。  
- 后端通过 `llm-gateway` 统一管理模型厂商差异。  
- 提示词模板版本化，支持灰度切换。  
- 模型不可用时可降级到规则引擎/历史结果缓存。  

### 22.2 推荐落点功能
- 简历/岗位文本结构化抽取（实体识别 + 标准化）。  
- 匹配解释文案生成（基于结构化结果，不直接信任原始生成文本）。  
- 学习建议生成（结合缺失技能与岗位要求）。  
- 管理端质检助手（解析失败原因归类建议）。  

### 22.3 网关接口设计（示例）
| 接口 | 方法 | 说明 |
|---|---|---|
| `/api/v1/llm/extract` | POST | 文本结构化抽取（内部调用为主） |
| `/api/v1/llm/explain-match` | POST | 生成匹配解释文案 |
| `/api/v1/llm/generate-suggestion` | POST | 生成学习/改进建议 |

统一错误体建议：
```json
{
  "code": "LLM_TIMEOUT",
  "message": "model response timeout",
  "requestId": "trace-xxx"
}
```

### 22.4 安全与成本控制
- 每用户/每租户限流，防止滥用与成本失控。  
- 对提示词与响应做审计留痕（脱敏后存储）。  
- 记录 token 消耗、耗时、成功率，形成成本看板。
